## Overview

Terraform monorepo managing DNS records and zone settings for multiple Cloudflare domains. Each domain has isolated Terraform state stored in a Cloudflare R2 bucket (S3-compatible backend).

## Claude Skills

Two slash commands are available for common workflows:

```
/new-domain example.com      # Scaffold a new domain and run terraform init
/import-domain example.com   # Import existing Cloudflare domain (runs script, init, imports, plan)
```

These handle the full end-to-end workflow including terraform init with the R2 backend endpoint.

## Common Commands

All Terraform commands must be run from within a domain's directory (e.g., `domains/example.com/`), not the repo root.

```sh
# Source credentials before any Terraform or script commands
source .env

# Initialize a domain workspace (required once per domain, and after any backend changes)
cd domains/example.com
terraform init -backend-config="endpoint=https://${TF_VAR_account_id}.r2.cloudflarestorage.com"

# Plan and apply changes
terraform plan
terraform apply
```

## Architecture

**Per-domain isolation**: Each domain in `domains/<domain>/` is a self-contained Terraform root module with its own backend state in R2. They all call the shared `modules/domain` module.

**Shared module** (`modules/domain/`): Looks up the Cloudflare zone by account ID + domain name, creates `cloudflare_record` resources via `for_each` over a list-to-map transformation, and manages zone settings via `cloudflare_zone_settings_override`.

**DNS record keying**: Records are keyed as `"${type}_${name}_${idx}"` — the index matters for stable plan output. Reordering records in `terraform.tfvars` will trigger resource replacements.

**All configuration lives in `terraform.tfvars`**: The `main.tf` and `variables.tf` files in each domain directory are boilerplate. Day-to-day changes only touch `terraform.tfvars`.

**Backend note**: The R2 endpoint is not stored in `main.tf` (it contains the account ID). It must be passed via `-backend-config` flag during `terraform init`. If init was run without the endpoint set correctly, re-run with `-reconfigure`.

## Environment Variables

All five variables must be exported before running Terraform. Copy `.env.example` to `.env` and fill in your values (`.env` is gitignored).

| Variable | Description |
|---|---|
| `CLOUDFLARE_EMAIL` | Cloudflare account email |
| `CLOUDFLARE_API_KEY` | Global API Key |
| `AWS_ACCESS_KEY_ID` | R2 API token access key ID |
| `AWS_SECRET_ACCESS_KEY` | R2 API token secret key |
| `TF_VAR_account_id` | Cloudflare account ID |

The `AWS_*` vars use that naming because Terraform's S3 backend reads them automatically; R2 exposes an S3-compatible API.

## Provider Version

Pinned to Cloudflare provider `~> 4.0`. The module uses `cloudflare_zone_settings_override` which was removed in provider v5 — do not upgrade to v5 without migrating to individual `cloudflare_zone_setting` resources.
